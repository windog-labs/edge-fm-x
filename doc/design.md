# Design

## 设计背景

随着大语言模型和视觉语言模型在边缘设备上的部署需求增长，现有推理框架在边缘端场景下存在不足。本项目旨在设计一个专门针对边缘端优化的轻量级推理引擎。

### 现有框架的局限性

#### 1. 云端服务框架的复杂性

[vLLM](https://github.com/vllm-project/vllm)、[SGLang](https://github.com/sgl-project/sglang) 和 [TensorRT-LLM](https://github.com/NVIDIA/TensorRT-LLM) 等框架主要面向高吞吐量的云端服务场景，包含复杂的批处理调度、多租户管理、分布式推理等功能。这些设计虽然适合云端服务，但对于边缘端的单请求、低延迟场景引入了不必要的复杂度和开销。

#### 2. llama.cpp 的兼容性负担

[llama.cpp](https://github.com/ggml-org/llama.cpp) 为了支持广泛的模型架构和硬件平台（CPU、GPU、移动设备等），代码库变得庞大复杂。同时为了提供服务能力，需要依赖 HTTP 服务器等组件，增加了部署复杂度。这种"通用性优先"的设计理念，使得框架在特定场景下的优化空间受限。

#### 3. 边缘端特殊场景的缺失优化

现有框架主要面向云端或通用场景，未充分考虑边缘端的特殊需求：单请求场景不需要复杂的批处理调度、小词表和小尺寸算子缺乏针对性优化、内存受限场景需要更激进的 KV Cache 压缩策略。

### 设计目标

基于以上分析，EdgeFM 的设计目标：

1. **轻量级设计**：去除批处理、多租户等复杂功能，专注于单请求、低延迟场景
2. **边缘端优化**：针对小词表、小尺寸算子、内存受限等特点进行专门优化
3. **简洁的代码结构**：保持代码库精简，便于维护和定制
4. **高效的 KV Cache 管理**：实现适合边缘端内存约束的压缩策略

### 模块说明

#### 1. 核心接口层 (`edge-fm.cpp`)
- **EdgeFM**: 对外提供的主要接口类，封装了完整的推理流程
- 使用 PIMPL 模式隐藏实现细节

#### 2. 推理引擎层 (`engine/`)
- **Engine**: 引擎基类，定义了推理的基本流程
  - **StandardEngine**: 标准推理引擎实现，支持多阶段量化模型和小词表优化
  - **EagleEngine**: 投机采样引擎（位于 `speculative/` 子目录），支持投机解码加速、多阶段量化模型和小词表优化
- **KVManager**: 管理 KV Cache 的分配、释放和压缩
  - 单请求提前分配：针对边缘端单请求场景，提前为每个请求分配固定大小的 KV Cache
  - KV 压缩算法：实现适合边缘端内存约束的压缩策略
- **Scheduler**: 负责请求调度和序列管理
  - 单请求匹配：针对单请求场景的简化调度逻辑，无需复杂的批处理匹配

#### 3. 模型层 (`models/`)
- **Model**: 模型基类，定义了 `prefill` 和 `decode_step` 接口
- **Qwen2.5**: Qwen2.5-VL 模型的具体实现

#### 4. 网络层 (`layers/`)
包含注意力机制、线性层、归一化层、激活函数、位置编码、嵌入层和采样器等基础神经网络组件。

**核心优化特性：**
- **特定尺寸算子优化**：针对边缘端模型的特殊尺寸（如小激活值、小词表等）进行专门的算子优化
- **前沿高性能算子集成**：集成 FlashInfer、SageAttention、MLA（Multi-head Latent Attention）等前沿高效算子
- **词表压缩实现**：实现词表压缩策略，减少内存占用和计算开销

#### 5. 工具层 (`utils/`)
提供配置管理、日志系统、性能指标和性能分析等工具功能。

#### 6. Python 绑定层 (`python/`)
- **pybind.cpp**: 使用 pybind11 提供 Python 接口
- 绑定核心类型：`EdgeFM`、`Tensor`、`Request`、`Response` 等
